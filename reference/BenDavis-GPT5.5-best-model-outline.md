## 一、总体评价

### 1.1 使用背景
- 已使用 GPT-5.5 约四周(Early Access)
- 主要在使用 Codex(low reasoning)和 ChatGPT Pro 的 X-High 推理

### 1.2 核心观点
- 这是"目前最好的模型"
- 新预训练使得基础模型质量大幅提升
- "能够做那些在 GPT-5 Codex 上做不到的事"

---

## 二、推理模式选择

### 2.1 Low Reasoning 推荐
- Ben Davis 强推 low/no reasoning 模式
- 在 UI/设计/前端任务上表现最佳
- 不浪费 token 去过度思考

### 2.2 X-High Reasoning 场景
- 仅用于复杂数学/逻辑问题
- 在大多数编码场景中不需要

### 2.3 GPT-5.4 vs GPT-5.5 推理对比
- GPT-5.4 在 low 推理下不稳定
- GPT-5.5 即使是 low 推理也已经很智能
- 新预训练让非推理模式就已经很好用

---

## 三、Token 效率分析

### 3.1 定价变化
- GPT-5.5 定价是 GPT-5.4 的 **2 倍**
- $5/M 输入 token, $30/M 输出 token
- Pro 版: $30/$180

### 3.2 Token 节约
- 完成同样任务使用 **少 20-40% 的 token**
- 实际成本增加不大(效率提升部分抵消了涨价)
- 效果更好，总体性价比不降反升
- "输出 token 大幅度减少的同时，完成质量更好"

---

## 四、实际使用体验

### 4.1 代码质量
- 代码更"人性化"，不盲目追求大段代码
- 能保持全项目上下文
- 理解应该修改的最小范围

### 4.2 AI 的"克制"
- 不像以前那样过度设计/过度工程
- 能守住约束(例如：不修改不该改的代码)
- "更具战略性地使用代码，只做必要的改动"

### 4.3 Compaction(压缩)问题
- GPT-5.5 的 compaction 比 GPT-5.4 更弱
- 在长对话中更容易丢失上下文
- **建议把线程控制在 100K token 以内**
- 压缩引发上下文断裂的风险更高

---

## 五、工具和工作流推荐

### 5.1 Pi 作为 Harness
- Ben Davis 强烈推荐 Pi(而非 Codex CLI)
- Pi 处理上下文摘要更优
- 线程管理更稳定
- 支持工作树(worktree)功能，让多个分支同时运行

### 5.2 Pi 的优势
- 更好的上下文压缩和恢复
- 支持 GPT-5.5(Codex 暂不支持)
- 更积极的社区维护

### 5.3 工作树(Worktree)操作
- 可以对同一仓库创建多个并行分支
- 每个分支可以独立运行不同任务
- 显著提高效率

---

## 六、上下文窗口管理

### 6.1 窗口限制
- Codex 中 GPT-5.5 有 **400K 上下文窗口**
- 但实际表现建议在 100K 以内

### 6.2 敏感性
- GPT-5.5 对上下文窗口的敏感度高于 GPT-5.4
- 接近窗口上限时质量下降明显
- 需更积极管理长对话

### 6.3 最佳实践
- 保持线程短小精悍
- 完成一个任务后开启新对话
- 避免在同一个线程中长时间工作

---

## 七、OpenAI 的 API 策略

### 7.1 当前状态
- API 尚未开放 GPT-5.5
- 延迟原因：等待更多防护措施就位(yet pending safeguards)
- Codex 已支持，但 ChatGPT API 路线不明

### 7.2 推测
- 可能将 GPT-5.5 优先绑定到自家平台(Codex)
- 第三方 API 可能需等待更长时间
- 策略转变：从模型公司变成 AI 推理公司(Altman 原话)

---

## 八、关键结论

1. GPT-5.5 是目前最好的模型，尤其推荐 low/no reasoning 模式
2. 价格翻倍但 token 效率的提升大多抵消了额外成本
3. 上下文窗口管理是新问题 — compaction 变弱了
4. Pi 是当前最佳的 GPT-5.5 harness 选择
5. API 用户需要等待更长时间才能使用
